## LLM-to-LLM Collaboration Workflow (Requirements Elicitation + Handoff Protocol)

### Purpose
A lightweight, repeatable protocol for two (or more) LLMs to coordinate on a task with minimal ambiguity, fast convergence on requirements, and clean handoffs between phases/agents.

### Roles (recommended)
- **Coordinator (C):** drives elicitation, maintains canonical “task state,” resolves ambiguity, enforces constraints, produces final integrated output.
- **Specialist (S):** provides domain-specific content, alternatives, edge cases, critique, verification, or implementation details.
- **Reviewer (R) (optional):** performs adversarial review (correctness, safety, completeness), suggests fixes.

---

## Canonical Task State (single source of truth)
Maintain a shared “task state” object, updated each turn. Keep it stable and versioned.

**Task State fields**
1) **Goal**: one-sentence objective  
2) **Audience / use case**  
3) **Deliverable(s)**: format(s), structure, language  
4) **Constraints**: length, tone, tools, citations, safety/policy constraints  
5) **Inputs**: raw material, links, data, APIs (if any)  
6) **Assumptions**: explicit defaults being used  
7) **Acceptance criteria**: what counts as “done” (tests, checklist, rubric)  
8) **Open questions**: prioritized, smallest set needed to proceed  
9) **Plan**: steps + owner per step (C/S/R)  
10) **Progress log**: what’s completed, what’s next  
11) **Risks / uncertainties**: known gaps, verification limits

**Rule:** If a field is unknown, mark it `TBD` and decide whether it blocks progress.

---

## Workflow Phases

### Phase 0 — Kickoff (1 turn)
**Goal:** establish the minimal viable task state.
- C asks for: goal, audience, deliverable, constraints, existing artifacts.
- S can propose candidate goals if goal is missing, but should request a selection.

**Exit criteria:** Goal + deliverable type + major constraints are set (even if provisional).

---

### Phase 1 — Requirements Elicitation (1–3 turns)
**Goal:** converge on clarity with minimal questions.

**Technique: “Blockers-first interrogation”**
Ask only questions that (a) change the output materially or (b) prevent starting. Typical blockers:
- Required format/sections?
- Strict length?
- External tools allowed (browsing/code exec)?
- Citation requirements / source constraints?
- Must-follow style guide?
- Intended user action after reading?
- Definition of correctness (tests, acceptance checklist)?

**Rule of 3:** Ask at most 3–5 questions per turn; propose defaults for the rest.

**Exit criteria:** Acceptance criteria and plan are agreed; remaining unknowns are non-blocking.

---

### Phase 2 — Task Decomposition + Assignment (1 turn)
**Goal:** assign work to agents without overlap.
- C produces a short plan with subtasks and owners.
- S confirms scope, adds edge cases/risks, proposes alternatives.

**Exit criteria:** Each subtask has (owner, expected output, deadline/turn).

---

### Phase 3 — Execution (N turns)
**Goal:** produce artifacts with predictable structure.

**Per-turn structure (recommended)**
- **State delta:** what changed in task state
- **Output:** the artifact (or a chunk)
- **Verification:** quick self-check against acceptance criteria
- **Next:** what the other agent should do next

**Chunking rule:** prefer complete sections over partial paragraphs; minimize cross-references.

---

### Phase 4 — Review / Integration (1–2 turns)
**Goal:** ensure internal consistency and constraint compliance.
- R/S checks: missing requirements, contradictions, length, tone, safety.
- C integrates, resolves conflicts, updates final task state.

**Exit criteria:** acceptance criteria met; no open questions that block delivery.

---

### Phase 5 — Handoff / Packaging (final turn)
**Goal:** deliver final output + reusable state for continuation.
Include:
- Final deliverable
- Final task state snapshot
- Decision log (key choices + rationale)
- Known limitations + suggested next steps

---

## Handoff Protocol (what to send when switching agents)
When passing to another LLM (or resuming later), send a compact “handoff bundle”:

1) **Task State (current version)**  
2) **What’s done (bullet list)**  
3) **What remains (prioritized)**  
4) **Constraints reminders** (length/tools/citations)  
5) **Open questions** (only those that still matter)  
6) **Files/artifacts** (embedded or referenced)  
7) **Quality concerns** (uncertainties, spots needing verification)

---

## Failure Modes + Mitigations (common in LLM-to-LLM)
- **Ambiguous ownership / duplicated effort** → assign owners per subtask; define deliverable boundaries.
- **State drift (agents disagree on requirements)** → enforce single canonical task state; every change is explicit.
- **Over-questioning** → cap questions/turn; propose defaults; proceed with assumptions labeled.
- **Tooling mismatch** (one assumes browsing) → record tool availability in constraints; forbid unverified citations.
- **Integration contradictions** → add a review phase; require consistency check (terminology, definitions, formatting).

### `python/validate_task_state.py`
```python
import json, sys
from jsonschema import Draft202012Validator

SCHEMA_PATH = "../schemas/canonical-task-state.schema.json"

with open(SCHEMA_PATH, "r", encoding="utf-8") as f:
    schema = json.load(f)

validator = Draft202012Validator(schema)

def validate_task_state(obj: dict):
    errors = sorted(validator.iter_errors(obj), key=lambda e: list(e.path))
    if errors:
        return False, [e.message for e in errors]
    return True, []

if __name__ == "__main__":
    with open(sys.argv[1], "r", encoding="utf-8") as f:
        obj = json.load(f)
    ok, errs = validate_task_state(obj)
    print(json.dumps({"ok": ok, "errors": errs}, indent=2))
    sys.exit(0 if ok else 1)
```
